# Fenjue V2 Migration and Rollback

## 范围

本目录描述从现有原型表迁移到 `schema_v2.sql` 的办法。它是部署规格，不会自动修改极空间上的 Hermes 数据库。迁移前必须取得实际数据库副本并完成字段盘点；禁止对未知生产库直接执行通用 `ALTER TABLE`。

## 为什么不能原地改列

V2 同时增加外键、`CHECK`、作用域字段，并把价格/金额从 `REAL` 改为整数最小单位。SQLite 对约束和列类型的原地修改能力有限，因此采用“新表—校验—切换”的影子迁移。

## 上行迁移

1. 停止 Hermes 的数据库写入任务，行情抓取可继续写入旧库或暂存队列。
2. 复制数据库、Skill 目录、Docker Compose/环境变量和定时任务配置，并记录 SHA-256。
3. 运行兼容性探针，确认 SQLite 版本、`foreign_keys=1`；JSON1/`STRICT` 仅记录能力，不作为基础迁移前提。
4. 在独立数据库执行 `schema_v2.sql`，不得直接覆盖旧表。
5. 按依赖顺序转换数据：账户 → 持仓 → 批次 → 风险配置 → 原始事件 → 特征/决策 → 意图/结果 → 实验。
6. 金额用十进制定点转换：人民币 × 100 后写 `*_fen`；每股价格 × 10000 后写 `*_price_x10000`。每批转换保存行数、总数量、总金额和哈希。
7. 为缺失逻辑簇的数据写入 `UNCLASSIFIED:<mapping_version>`，不得留空；这些记录不参加独立簇概率发布。
8. 旧的 `retired_leakage` 转换为 `status='retired'`、`retire_reason='leakage'`。
9. 风险配置补齐账户和作用域；不能证明作用域的旧配置不迁入生产有效区间。
10. 执行 `integrity_checks.sql`、账本对账和固定夹具回放。任何违规都终止切换。
11. 将应用切换到 V2 数据库，先只读观察，再启用追加写；影子运行连接保持持仓写禁止。
12. 保留旧数据库和旧 Skill，不做删除，直到候选生产版完成观察期。

## 必须保存的迁移清单

- 迁移 ID、开始/完成时间、操作者和源/目标数据库哈希。
- 每张表转换前后行数。
- 每个账户的总股数、可卖股数、核心仓下限和权益总额差异。
- 丢弃、隔离、无法分类及四舍五入的全部记录。
- 使用的转换代码哈希、交易日历版本和逻辑簇映射版本。
- 完整性检查输出和失败原因。

## 回滚

回滚触发条件：账本不一致、时间泄漏、外键失败、来源时间异常、核心仓违规或生产性能不可接受。

1. 立即关闭 V2 动作开关，切换为 `OBSERVE`；事件冻结仍保持安全生效。
2. 停止 V2 写入并保存目标库副本，不删除失败证据。
3. 将 Hermes 指向迁移前只读校验通过的旧数据库和旧 Skill 配置。
4. 回放切换后发生的用户实际成交；不得从影子决策推导成交。
5. 对比账户、批次和人工覆盖，确认旧路径恢复后再恢复提示服务。
6. 记录回滚事故；修复后使用新的迁移 ID 从源快照重新开始，禁止在失败目标库上继续补丁式推进。

## 不可逆边界

- 已产生的用户实际成交无法通过数据库回滚撤销，只能重新对账。
- V2 的原始快照、决策、拒绝、冻结解除和实验记录属于审计证据，不应删除。
- 从整数单位回写旧 `REAL` 仅用于旧系统兼容展示，不再作为权威账本。

## 正式执行前需要补齐

取得极空间生产数据库副本后，再生成针对实际旧表的：

- `preflight_inventory.sql`
- `up_<migration_id>.sql`
- `verify_<migration_id>.sql`
- `rollback_<migration_id>.sql`

这四个文件必须在副本上演练两次，并留下逐表校验结果，才允许进入维护窗口。
